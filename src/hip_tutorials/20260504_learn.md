## 1. MPI 基礎概念

### 什麼是 MPI

**MPI**（Message Passing Interface，訊息傳遞介面）是一套**規範**，用來讓多個獨立執行的程序（process）之間互相傳遞資料、協調運算。

- 它**不是程式語言**，而是一組 API 規範
- 主要實作：**OpenMPI** 和 **MPICH**
- 用途：**分散式記憶體並行運算**的標準溝通方式
- 場景：HPC 高效能運算叢集、超級電腦

### 為什麼需要 MPI

當運算量大到一台機器、一顆 CPU、甚至一張 GPU 都跑不完時，需要把工作切開分給多個運算單元：

| 場景 | 解法 |
|---|---|
| 同一台機器內多執行緒 | OpenMP、pthread、CUDA/HIP threads（共用記憶體）|
| **跨機器、跨節點**（分散式記憶體）| **MPI**（透過訊息傳遞）|

### 核心概念

啟動 MPI 程式時（`mpirun -n 4 ./my_program`），會同時啟動 N 個一模一樣的 process：

| 概念 | 說明 |
|---|---|
| **Rank** | 每個 process 的編號（0, 1, 2, ..., N-1）|
| **Size** | 總共有幾個 process |
| **Communicator** | 一群可以互相通訊的 process 集合，預設 `MPI_COMM_WORLD` |
| **Point-to-point 通訊** | `MPI_Send` / `MPI_Recv`，一對一傳訊 |
| **Collective 通訊** | `MPI_Bcast`、`MPI_Reduce`、`MPI_Allreduce`、`MPI_Scatter`、`MPI_Gather` |

## 2. MPI Hello World

### 範例程式碼

```c
#include <mpi.h>
#include <stdio.h>

int main(int argc, char** argv) {
    MPI_Init(&argc, &argv);                         // ① 啟動 MPI 環境

    int world_size;
    MPI_Comm_size(MPI_COMM_WORLD, &world_size);     // ② 總共幾個 process

    int world_rank;
    MPI_Comm_rank(MPI_COMM_WORLD, &world_rank);     // ③ 我是第幾號

    printf("Hello world from rank %d out of %d processors\n",
           world_rank, world_size);

    MPI_Finalize();                                  // ④ 收尾、釋放資源
}
```

### MPI 程式的標準骨架

| 步驟 | 函式 | 作用 |
|---|---|---|
| ① | `MPI_Init` | 開場：建立通訊環境 |
| ② | `MPI_Comm_size` | 問：「我們這團一共幾個人？」|
| ③ | `MPI_Comm_rank` | 問：「我在這團的編號是？」|
| ④ | `MPI_Finalize` | 收場：清理環境 |

### 編譯與執行

```bash
mpicc hello.c -o hello       # 用 MPI 專用的編譯器包裝
mpirun -n 4 ./hello          # 啟動 4 個 process
```

輸出順序**不固定**（4 個 process 同時跑）。

> ⚠️ **重要術語修正**：MPI 管的是 **process**（行程），不是 thread（執行緒）。每個 MPI process 有自己獨立的記憶體空間——這是 MPI 跟 OpenMP 最關鍵的差別。

## 3. MPI 同步與歸約

### 為什麼會「完成時間不一樣」

不同 process 完成同一段工作時間可能不同，原因：

- **工作量不均**：每個 rank 拿到的資料大小、計算複雜度不同
- **硬體差異**：不同節點微小差異、其他程式搶資源
- **隨機性**：演算法本身的隨機性
- **網路延遲**：跨節點通訊有抖動
- **OS 排程**：作業系統可能讓 CPU 給別的 process

### MPI_Barrier 同步

```c
MPI_Barrier(MPI_COMM_WORLD);
```

語意：**「所有 process 都到達這一行之前，誰都不准往下走。」**

```
時間 →

rank 0:  ████████░░░░░░░░░ [BARRIER] ████████
rank 1:  █████░░░░░░░░░░░░ [BARRIER] ████████
rank 2:  ██████████████░░░ [BARRIER] ████████
rank 3:  ███████░░░░░░░░░░ [BARRIER] ████████
                            ↑
                   全部到齊後才一起出發
```

### ⚠️ Barrier 其實很少需要主動用

**大部分 MPI 通訊函式本身就帶有同步效果**：

| 函式 | 是否隱含同步 |
|---|---|
| `MPI_Bcast` | ✅ 收的人會等發的人 |
| `MPI_Reduce` / `MPI_Allreduce` | ✅ 全部人都要參與才會完成 |
| `MPI_Recv`（阻塞式）| ✅ 收不到就一直等 |
| `MPI_Gather` / `MPI_Scatter` | ✅ 同上 |

**真的會用到 Barrier 的場合**：
1. **效能量測 / 計時**：前後加 Barrier 確保大家同時開始/結束
2. **Debug / 印訊息**：讓輸出比較整齊
3. **跟 MPI 之外的東西同步**：例如等所有 process 都把檔案寫完

> 💡 HPC 老話：**"If you think you need a barrier, you probably don't."**
> Barrier 會傷效能——強迫快的人停下來等慢的人。

### MPI_Reduce 範例

```c
MPI_Bcast(&start_number, 1, MPI_INT, 0, MPI_COMM_WORLD);

int random_number = rand_r(&seed) % 100;
start_number += random_number;

MPI_Barrier(MPI_COMM_WORLD);   // ⚠️ 教學用，實務上不需要

MPI_Reduce(&start_number, &end_number, 1,
           MPI_INT, MPI_SUM, 0, MPI_COMM_WORLD);

if (world_rank == 0) {
    printf("The average number is %d\n", end_number / world_size);
}
```

### MPI_Reduce 參數詳解

```c
MPI_Reduce(&start_number,   // ① 來源：每個 rank 自己的值
           &end_number,     // ② 目的地：只有 root 收得到
           1,               // ③ 元素數量
           MPI_INT,         // ④ 資料型態
           MPI_SUM,         // ⑤ 運算方式
           0,               // ⑥ root rank：誰來收結果
           MPI_COMM_WORLD); // ⑦ 通訊群組
```

### Reduce 運算選項

| 運算 | 意義 |
|---|---|
| `MPI_SUM` | 加總 |
| `MPI_PROD` | 連乘 |
| `MPI_MAX` / `MPI_MIN` | 最大/最小值 |
| `MPI_MAXLOC` / `MPI_MINLOC` | 最大/最小值**及其位置** |
| `MPI_LAND` / `MPI_LOR` | 邏輯 AND / OR |

### 為什麼 printf 要包在 if (rank == 0) 裡

1. **只有 rank 0 拿到完整結果**：Reduce 的結果只送到 root，其他 rank 的 `end_number` 是垃圾值
2. **避免重複輸出**：不加 if 會出現 N 行訊息

> 這是 MPI 的常見模式：**「大家一起算，但只有一個人負責報告結果。」**

## 4. HIP 雙軌架構：Host 與 Device

### 一份原始碼，兩個世界

**最關鍵的觀念**：寫一份 `.cpp` 檔，但裡面同時包含兩種程式碼，最後跑在兩個不同的硬體上。

```
        HIP Program (.cpp)
       ┌──────────────────┐
       │  // CPU 部分      │
       │  int main() {...} │  ──→ 編譯給 CPU 執行
       │                  │
       │  // GPU 部分      │
       │  __global__ ...  │  ──→ 編譯給 GPU 執行
       └──────────────────┘
```

### Host 端（CPU 世界）

| 層級 | 角色 |
|---|---|
| **CPU Code** | 一般 C++ 程式碼，邏輯控制、檔案 I/O |
| **CPU "Host"** | 執行硬體（Intel/AMD CPU）|
| **CPU/System Memory** | 主記憶體（DRAM）|

**Host 是「指揮官」**：讀檔、配置 GPU 記憶體、搬資料、發射 kernel、收結果。

### Device 端（GPU 世界）

| 層級 | 角色 |
|---|---|
| **GPU Code** | 用 `__global__` 標記的 kernel |
| **GPU "Device"** | 執行硬體（AMD Instinct / Radeon）|
| **GPU/Graphics Memory** | 顯示卡上的 VRAM（HBM2/HBM3 或 GDDR6）|

**Device 是「大量計算的工人」**：同時開幾萬個 thread 平行運算，只能存取自己的記憶體。

### ⚠️ 關鍵痛點：兩邊記憶體分開

```
   CPU 看不到 GPU 記憶體 ❌
   GPU 看不到 CPU 記憶體 ❌

   [CPU 記憶體] ←──PCIe 匯流排──→ [GPU 記憶體]
                  必須手動搬運！
```

### 正確的資料流程

```cpp
// 1. CPU 端準備資料
int* h_data = (int*)malloc(N * sizeof(int));   // h_ = host
for (int i = 0; i < N; i++) h_data[i] = i;

// 2. 在 GPU 上配置記憶體
int* d_data;                                    // d_ = device
hipMalloc(&d_data, N * sizeof(int));

// 3. 把資料從 CPU 搬到 GPU
hipMemcpy(d_data, h_data, N * sizeof(int), hipMemcpyHostToDevice);

// 4. 發射 kernel
my_kernel<<<blocks, threads>>>(d_data, N);

// 5. 把結果從 GPU 搬回 CPU
hipMemcpy(h_data, d_data, N * sizeof(int), hipMemcpyDeviceToHost);

// 6. 釋放 GPU 記憶體
hipFree(d_data);
free(h_data);
```

> 💡 **命名慣例**：用 `h_` 前綴表示 host pointer，`d_` 前綴表示 device pointer。

### 函式修飾詞

| 修飾詞 | 在哪裡執行 | 從哪裡呼叫 |
|---|---|---|
| `__host__`（預設）| CPU | CPU |
| `__global__` | **GPU** | **CPU**（kernel 入口！）|
| `__device__` | GPU | GPU（GPU 內部 helper）|
| `__host__ __device__` | 兩邊都編譯 | 兩邊都可呼叫 |

## 5. HIP 編譯流程與 Runtime

### HIP 編譯：一份檔案、兩次編譯

HIP 編譯器 `hipcc`（背後是 LLVM/Clang）做的事叫 **"single-source, dual-compilation"**。

```
                 my_program.cpp
                 ┌────────────────────┐
                 │  __global__ kernel │
                 │  int main() {...}  │
                 └─────────┬──────────┘
                           │
                      hipcc 解析
                           │
              ┌────────────┴────────────┐
              ▼                         ▼
       ┌──────────────┐          ┌──────────────┐
       │  Host 部分    │          │ Device 部分   │
       └──────┬───────┘          └──────┬───────┘
              │                         │
         Clang (x86)                Clang (AMDGPU)
              │                         │
              ▼                         ▼
       x86-64 機器碼              GCN/RDNA ISA
              │                         │
              └────────────┬────────────┘
                           ▼
                  ┌─────────────────┐
                  │ 單一可執行檔      │
                  │  ┌───────────┐  │
                  │  │ CPU 機器碼 │  │
                  │  ├───────────┤  │
                  │  │ GPU 機器碼 │  │ ← 嵌入在執行檔裡（fat binary）
                  │  └───────────┘  │
                  └─────────────────┘
```

### Runtime 的兩種意思

#### 意思 1：時間概念

**Runtime = 程式執行的那段時間**，相對於 compile-time。

```
寫 code  →  編譯  →  ┃  執行  ┃  →  結束
                    ┗━━━━━━━┛
   compile-time          runtime
```

#### 意思 2：函式庫

**「程式執行時，背後支援它運作的那一套函式庫和系統。」**

| 你寫的 | 背後 runtime 幫你做的事 |
|---|---|
| `printf("hi")` | C runtime 把字串送到 OS 的 stdout |
| `new int[100]` | C++ runtime 跟 OS 要記憶體 |
| `hipMalloc(...)` | **HIP runtime** 跟 GPU 驅動要 VRAM |

### HIP Runtime 在做什麼

```cpp
hipMalloc(&d_data, 1024);                              // 跟 GPU 驅動要 VRAM
hipMemcpy(d_data, h_data, 1024, hipMemcpyHostToDevice); // 透過 PCIe / DMA 搬資料
my_kernel<<<10, 256>>>(d_data);                         // 載入 GPU 機器碼、啟動
hipDeviceSynchronize();                                 // 等 GPU 完成
hipFree(d_data);                                        // 釋放 GPU 記憶體
```

### HIP 軟體堆疊

```
┌───────────────────────────────────┐
│   你的應用程式                      │  ← 你寫的
├───────────────────────────────────┤
│   HIP Runtime API                 │  ← hipMalloc, hipMemcpy...
│   (libamdhip64.so)                │     這就是「HIP runtime」
├───────────────────────────────────┤
│   ROCm Runtime (HSA, ROCr)        │  ← 更底層的硬體抽象層
├───────────────────────────────────┤
│   AMD GPU Driver (amdkfd, amdgpu) │  ← Linux kernel driver
├───────────────────────────────────┤
│           AMD GPU 硬體             │  ← 真正的晶片
└───────────────────────────────────┘
```

### Mental Model

把 HIP runtime 想成**「派遣 GPU 工作的客服中心」**：
- 你打電話：「我要 1MB GPU 記憶體」→ HIP runtime 處理
- 你打電話：「把這份資料傳到 GPU」→ HIP runtime 處理
- 你打電話：「跑這個 kernel」→ HIP runtime 處理

不需要直接跟驅動程式講話、不用管 PCIe 怎麼運作——**runtime 就是把麻煩事抽象掉的中介層**。

## 6. GPU 與分散式架構的整合

### APU 觀念

**APU（Accelerated Processing Unit）** 是 AMD 商標（2011 起），但「**CPU+GPU 整合**」是業界共通趨勢：

| 廠商 | 名稱 | 例子 |
|---|---|---|
| **AMD** | APU | Ryzen 5600G、MI300A、PS5/Xbox 主機晶片 |
| **Intel** | iGPU / Xe | Lunar Lake、Ponte Vecchio |
| **Apple** | Apple Silicon | M1/M2/M3/M4（Unified Memory）|
| **NVIDIA** | Grace Hopper / Grace Blackwell | GH200、GB200 |

### APU 的權衡

| 瓶頸 | 獨立顯卡 | APU |
|---|---|---|
| **CPU↔GPU 傳輸** | 慢（PCIe 4.0 ~64GB/s）| **快**（共用記憶體）✅ |
| **GPU 記憶體頻寬** | **快**（HBM3 ~5TB/s）✅ | 慢（DDR5 ~100GB/s）❌ |
| **GPU 算力規模** | 大 ✅ | 受封裝限制 ❌ |

**APU 主場**：行動裝置、遊戲主機、HPC 超級電腦（如 El Capitan 用的 MI300A）。
**獨顯主場**：AI 訓練、大型科學運算、桌機高效能。

### MPI + HIP 的整合架構

業界主流：**「1 個 MPI rank 配 1 顆 GPU」**

```
假設一個節點：64 個 CPU 核心 + 4 顆 GPU

Node 0:
  ├─ MPI rank 0 → 綁定 CPU 核心 0~15  → 操作 GPU 0
  ├─ MPI rank 1 → 綁定 CPU 核心 16~31 → 操作 GPU 1
  ├─ MPI rank 2 → 綁定 CPU 核心 32~47 → 操作 GPU 2
  └─ MPI rank 3 → 綁定 CPU 核心 48~63 → 操作 GPU 3
```

剩餘 CPU 核心用 **OpenMP** 開多執行緒吃滿，這叫 **MPI + X** 三層並行：

```
┌─ 跨節點：MPI 通訊
│
├─ 節點內 CPU 計算：OpenMP 多執行緒
│
└─ 節點內 GPU 計算：HIP / CUDA kernel
```

### 程式中決定 GPU 對應

```cpp
int world_rank;
MPI_Comm_rank(MPI_COMM_WORLD, &world_rank);

int num_gpus;
hipGetDeviceCount(&num_gpus);

// 用 rank 對 GPU 數量取模
int my_gpu = world_rank % num_gpus;
hipSetDevice(my_gpu);

// 之後這個 rank 的所有 hipMalloc / kernel 都會跑在 my_gpu 上
```

### 啟動參數範例

```bash
# 2 個節點，每節點 4 個 rank（一 rank 一 GPU）
mpirun -n 8 -ppn 4 --bind-to numa ./my_program

# SLURM (超級電腦常見)
srun -N 100 --ntasks-per-node=4 --gpus-per-task=1 ./my_program
```

### GPU-aware MPI

當「1 rank 1 GPU」時，rank 之間的通訊本質上就是 **GPU 之間的通訊**：

```cpp
// 傳統作法（慢）：
hipMemcpy(h_buf, d_send_buf, size, DeviceToHost);
MPI_Send(h_buf, ...);
// 對方：MPI_Recv → hipMemcpy HostToDevice

// GPU-aware MPI（快）：
MPI_Send(d_send_buf, ...);  // 直接傳 GPU 指標！
// 底層走 GPUDirect / RDMA，繞過 CPU
```

### 與 ASTRA-sim 的連結

ASTRA-sim 模擬大規模 AI 訓練，它的 **host trace** 跟 **device trace** 正是這個 host/device 概念的延伸：

| Trace 類型 | 記錄內容 |
|---|---|
| **Host trace** | CPU 端：發射 kernel、配置/釋放 GPU 記憶體、發起 collective 通訊 |
| **Device trace** | GPU 端：kernel 實際執行、GPU 記憶體操作、kernel 之間依賴 |

兩條時間軸要分開記錄，效能分析才看得出 **CPU↔GPU 是否 overlap**——這是大規模訓練優化的核心。

## 7. HIP Kernel Launch

### Hello World Kernel

```cpp
#include <hip/hip_runtime.h>

__global__ void gpuHello() {
    int tid = blockIdx.x * blockDim.x + threadIdx.x;
    printf("Hello World from thread %d\n", tid);
}

int main() {
    gpuHello<<<4, 4>>>();
    hipDeviceSynchronize();
    return 0;
}
```

### Kernel 三大規則

1. **`__global__` 修飾詞**：函式跑在 GPU、從 CPU 呼叫
2. **必須回傳 `void`**：幾千個 thread 同時執行，回傳值無意義
3. **要傳結果只能用指標寫入 GPU 記憶體**

### Thread ID 計算

```cpp
int tid = blockIdx.x * blockDim.x + threadIdx.x;
```

| 變數 | 意義 |
|---|---|
| `threadIdx.x` | 我在 block 內的編號 |
| `blockIdx.x` | 我所在的 block 在 grid 內的編號 |
| `blockDim.x` | 一個 block 有幾個 thread |

### Triple Chevron 語法

```cpp
gpuHello <<< 4, 4 >>> ();
//          ↑  ↑
//          │  └── 每個 block 有幾個 thread
//          └───── 一共幾個 block
```

`<<<...>>>` 是 HIP/CUDA 對 C++ 的**語法擴展**，一般 C++ 編譯器看不懂，必須用 `hipcc`。

### Grid / Block / Thread 階層

```
┌─────────────── Grid (整個發射) ─────────────────┐
│                                                 │
│  ┌─ Block 0 ─┐  ┌─ Block 1 ─┐                  │
│  │ T0 T1     │  │ T0 T1     │                  │
│  │ T2 T3     │  │ T2 T3     │                  │
│  └───────────┘  └───────────┘                  │
│                                                 │
│  ┌─ Block 2 ─┐  ┌─ Block 3 ─┐                  │
│  │ T0 T1     │  │ T0 T1     │                  │
│  │ T2 T3     │  │ T2 T3     │                  │
│  └───────────┘  └───────────┘                  │
└─────────────────────────────────────────────────┘
```

**為什麼分層？**
1. **Block 內可共享記憶體 / 同步** (`__syncthreads()`)
2. **Block 之間獨立** → 可被 GPU 排到不同計算單元（CU/SM）平行
3. **硬體限制**：一個 block 最多 1024 個 thread

### Kernel Launch 是非同步的

```cpp
gpuHello<<<4, 4>>>();  // CPU 把任務丟給 GPU 後立刻 走人
                       // 不等 GPU 算完
hipDeviceSynchronize(); // ← 必須有這行，CPU 才會停下來等 GPU
```

如果沒有 `hipDeviceSynchronize()`，main 可能在 GPU 還沒印完就結束。

> 💡 非同步是**效能設計**——讓 CPU 在 GPU 算的同時去做別的事（準備下一批資料、發起 MPI 通訊）。

### MPI vs HIP 比較

| 特性 | 普通 C++ | MPI | HIP |
|---|---|---|---|
| 並行單位 | 單執行緒 | Process（行程）| Thread（執行緒）|
| 並行數量 | 1 | 幾十~幾千 | **幾萬~幾百萬** |
| 識別「我是誰」 | 不需要 | `MPI_Comm_rank` | `blockIdx`, `threadIdx` |
| 記憶體模型 | 單一 | 各自獨立 | 階層共享 |
| 啟動方式 | 直接 main | `mpirun -n N` | `kernel<<<blocks, threads>>>` |

## 8. 錯誤處理與 nodiscard

### 遇到的 Warning

```
warning: ignoring return value of function declared with 'nodiscard' attribute
   hipDeviceSynchronize();
```

### 原因

`hipDeviceSynchronize()` 回傳 `hipError_t` 錯誤碼，並標記為 `[[nodiscard]]`：

```cpp
[[nodiscard]] hipError_t hipDeviceSynchronize(void);
```

C++17 屬性 `[[nodiscard]]` 告訴編譯器：「**這個回傳值很重要，呼叫者忽略它就警告**」。

### 為什麼 GPU 程式特別需要錯誤檢查

GPU 程式有惡名昭彰的問題：**錯誤是非同步發生的**。

```cpp
my_kernel<<<999999, 999999>>>();  // 錯誤其實在這裡發生
                                  // 但 launch 是非同步的，CPU 不會立刻知道

hipDeviceSynchronize();  // ← 真正的錯誤在這裡才被「報告」
                         //   不檢查回傳值，永遠不知道出事了
```

### 解法 1：明確忽略

```cpp
(void)hipDeviceSynchronize();   // 告訴編譯器：我故意忽略
```

### 解法 2：檢查錯誤碼

```cpp
hipError_t err = hipDeviceSynchronize();
if (err != hipSuccess) {
    fprintf(stderr, "hipDeviceSynchronize failed: %s\n",
            hipGetErrorString(err));
    return 1;
}
```

### 解法 3：HIP_CHECK 巨集（業界標準）✅

```cpp
#define HIP_CHECK(cmd) do {                              \
    hipError_t err = cmd;                                \
    if (err != hipSuccess) {                             \
        fprintf(stderr, "HIP error: %s at %s:%d\n",      \
                hipGetErrorString(err), __FILE__, __LINE__); \
        exit(1);                                         \
    }                                                    \
} while(0)

// 使用：
HIP_CHECK(hipDeviceSynchronize());
HIP_CHECK(hipMalloc(&d_data, size));
HIP_CHECK(hipMemcpy(d_data, h_data, size, hipMemcpyHostToDevice));
```

優點：
- 一行搞定
- 自動印出檔案、行號（`__FILE__`, `__LINE__`）
- 整個 codebase 風格一致

### 學習方法論：講義 vs 實際環境差異處理

| 情況 | 建議做法 |
|---|---|
| 講義程式碼**完全跑不起來** | **修正**，註解原因 + 講義版本 |
| 講義程式碼能跑但有 **warning** | **修正**，註解原因（本次情況）|
| 講義程式碼**有 bug 但能跑** | **修正**，註解 |
| 講義程式碼是**過時 API** | 修正成新 API，註解舊 API 名稱 |

> **核心原則**：目標不是「跟講義一模一樣」，而是「**理解講義要教的概念，並寫出能在你環境正確運行的版本**」。

## 9. Makefile 完整教學

### 為什麼需要 Makefile

**Make 是一個「依賴關係追蹤 + 條件執行」的工具**，核心邏輯：

> **只重做「有必要重做」的事。**

它根據檔案修改時間自動判斷：目標比依賴新 → 不用重做；依賴比目標新 → 重做。

### Makefile 核心語法（只有一條規則）

```makefile
目標: 依賴1 依賴2 依賴3
	要執行的指令
```

> ⚠️ **超重要**：指令前面必須是 **Tab（製表符）**，不能是空格！

### 最簡單的 Makefile

```makefile
hello: hello.cpp
	g++ hello.cpp -o hello
```

執行 `make` 時：
1. 看 `hello` 存不存在？不存在 → 執行 recipe
2. 已存在 → 比較 `hello.cpp` 是否比 `hello` 新？是 → 重做；否 → up to date

### 預設目標：第一個目標

`make` 不指定目標時，會跑 **Makefile 裡寫的第一個目標**。慣例上叫 `all`：

```makefile
all: hello       # ← 第一個目標 = make 預設

hello: hello.cpp
	g++ hello.cpp -o hello
```

### 變數

```makefile
CXX = g++
FLAGS = -O2 -std=c++17

hello: hello.cpp
	$(CXX) $(FLAGS) hello.cpp -o hello
```

慣例：
- `CXX` = C++ 編譯器（`CC` = C 編譯器）
- `CXXFLAGS` / `CFLAGS` = 編譯參數
- 全大寫變數名是 Make 社群慣例

### 自動變數

| 符號 | 意思 | 範例展開 |
|---|---|---|
| `$@` | 當前**目標**名稱 | `hello` |
| `$<` | 第一個**依賴**名稱 | `hello.cpp` |
| `$^` | **所有**依賴 | `hello.cpp utils.cpp` |

### .PHONY 標記

```makefile
.PHONY: all run clean
```

告訴 Make「這些目標**不是檔案名**」，永遠執行 recipe。否則如果目錄下剛好有同名檔案，Make 會以為目標已存在而不執行。

慣例：`all`、`clean`、`install`、`test`、`run` 都要列進 `.PHONY`。

### Order-only Prerequisite (`|` 符號)

```makefile
$(TARGET): $(SRC) | $(BIN_DIR)
```

`|` 之後是**順序依賴**：「需要存在，但不要因為它的時間戳變了就重新編譯」。

用於目錄這種「存在性 prerequisite」——避免無限重編。

### 實戰：當前使用的 Makefile

```makefile
HIPCC = hipcc
HIPCC_FLAGS = -O2 -std=c++17

BIN_DIR = bin
TARGET = $(BIN_DIR)/hello
SRC = hello.cpp

all: $(TARGET)

$(BIN_DIR):
	mkdir -p $(BIN_DIR)

$(TARGET): $(SRC) | $(BIN_DIR)
	$(HIPCC) $(HIPCC_FLAGS) $< -o $@

run: $(TARGET)
	./$(TARGET)

clean:
	rm -rf $(BIN_DIR)

.PHONY: all run clean
```

執行 `make` 的完整流程：
```
1. 讀 Makefile
2. 沒指定目標 → 跑 all
3. all 依賴 $(TARGET) = bin/hello
4. bin/hello 依賴 hello.cpp 和 bin/（order-only）
5. 確保 bin/ 存在 → 沒有就 mkdir
6. 比較 hello.cpp 跟 bin/hello 修改時間
   - hello.cpp 較新 → 執行 hipcc 編譯
   - bin/hello 較新 → 「up to date」結束
```

### Makefile 的層次

| 層次 | 場景 | 工具 |
|---|---|---|
| **層次 1**：手寫小 Makefile | 單檔案、教學範例（**現在這個**）| Make |
| **層次 2**：手寫中型 Makefile（含通用規則）| 10~50 檔案 | Make + `%` 萬用規則 |
| **層次 3**：自動產生 Makefile | 100+ 檔案、跨平台（如 ROCm、LLVM、PyTorch）| **CMake** + Ninja |
| **層次 4**：高階語言 build 工具 | 各語言生態 | cargo / go build / npm |

### Mental Model

把 Makefile 想成**一份食譜**：

| 食譜 | Makefile |
|---|---|
| 要做什麼菜 | 目標 |
| 需要哪些食材 | 依賴 |
| 怎麼做 | Recipe |
| 標準調味料分量 | 變數 |
| 「清理廚房」（不是菜）| `.PHONY` |

`make` 像個聰明的廚師：你叫他做菜，他會看食材夠不夠新鮮，只重做需要重做的部分。

## 10. 關鍵心智模型總結

### 完整流程圖

```
你的 .cpp 檔
    │
hipcc 編譯（compile-time）
    │
┌───┴────┐
▼        ▼
CPU 機器碼  GPU 機器碼
(main)    (gpuHello kernel)
    │        │
    └─ 打包成執行檔 ─┘
        │
    執行（runtime）
        │
    CPU 開始跑 main()
        │
    遇到 <<<4, 4>>>
        │
    呼叫 HIP runtime
        │
    把 kernel 載入 GPU、配置 thread
        │
    GPU 平行執行 16 次 gpuHello
        │
    CPU 在 hipDeviceSynchronize() 等
        │
    GPU 完成 → CPU 往下走
```

### 三層並行的世界觀

```
┌─────── Node 0 ───────┐    ┌─────── Node 1 ───────┐
│  CPU (MPI rank 0)    │    │  CPU (MPI rank 1)    │
│   ↕ HIP API          │    │   ↕ HIP API          │
│  GPU                 │←──→│  GPU                 │
└──────────────────────┘ MPI └──────────────────────┘
       ↑                              ↑
   單機內：HIP 管 CPU↔GPU         跨節點：MPI 管 CPU↔CPU（或 GPU↔GPU）
```

| 層次 | 工具 | 解決什麼 |
|---|---|---|
| 跨節點 | **MPI** | 多台機器之間協調 |
| 節點內 CPU | **OpenMP** | 一顆 CPU 內多執行緒 |
| 節點內 GPU | **HIP / CUDA** | 一顆 GPU 內幾萬個 thread |

### 核心觀念對照表

| 概念 | 重點 |
|---|---|
| **MPI process 進度不一致** | 是事實，但不一定要 Barrier 強制同步 |
| **通訊函式隱含同步** | 大部分情況靠它就夠 |
| **Host-Device 記憶體分離** | 必須手動搬運（hipMemcpy）|
| **Kernel 必須 void** | 用指標寫結果 |
| **Kernel launch 是非同步** | 必須 hipDeviceSynchronize 等 |
| **錯誤是非同步發生** | 必須檢查 hipError_t |
| **1 MPI rank = 1 GPU** | 業界主流對應方式 |
| **Make 只重做必要的事** | 依靠時間戳判斷 |

### 常用命令速查

```bash
# MPI
mpicc hello.c -o hello
mpirun -n 4 ./hello

# HIP
hipcc -O2 -std=c++17 hello.cpp -o hello

# Make
make           # 編譯
make run       # 編譯 + 執行
make clean     # 清理

# 大規模 (SLURM)
srun -N 100 --ntasks-per-node=4 --gpus-per-task=1 ./train
```

### HPC 老話精華

> **"If you think you need a barrier, you probably don't."**
> 你以為你需要 barrier，其實多半不需要——通訊函式隱含同步，亂加 barrier 只會傷效能。
