## 一、平行運算三層次

### ILP 為什麼比 TLP/DLP 難實現
- 平行性必須由**硬體自己從循序指令流中挖掘**(TLP/DLP 由程式設計師明示)
- 必須處理三種相依性:RAW、WAR/WAW、控制相依
- 需要極複雜硬體機制:branch predictor、ROB、register renaming、speculative execution 等
- 受限於 ILP Wall:即使理想條件,可萃取量也有限

### ILP 突破的效益問題
- **不會突飛猛進**,因為:
  1. 程式本身的 critical path 是天花板
  2. Amdahl 定律的限制
  3. Memory Wall——大部分時間在等記憶體
  4. 功耗/面積非線性成長
- 但對單執行緒效能、延遲敏感應用仍然重要(Apple M 系列就是好例子)
- 業界共識:走異質運算,把工作分派給 CPU/GPU/NPU

## 二、Process vs Thread

### Shared Memory 的問題
無隔離、無保護、位址衝突、同步困難、記憶體管理混亂

### Process = 隔離的單位
- 每個 process 有獨立**虛擬位址空間**(投影片中 PID 1 和 PID 2 都用 `0x10,000` 但對應不同實體位址)
- 透過 page table 映射
- 提供安全性、穩定性、簡化編程

### Thread = 在 process 內提供平行執行的單位
- **共享**位址空間、heap、file descriptors
- **獨立**stack、暫存器、PC
- 建立/切換成本低,溝通快
- 代價:需要自行同步

### 關鍵對照
| 面向 | Process | Thread |
|---|---|---|
| 主要目的 | 隔離 | 共享 + 平行 |
| 位址空間 | 獨立 | 共享 |
| 建立成本 | 高 | 低 |

## 三、CPU Thread vs GPU Thread(同名但完全不同)

| 面向 | CPU Thread | GPU Thread |
|---|---|---|
| 數量 | 數百~數千 | 數萬~數百萬 |
| Context Switch | 高成本(μs) | **零成本**(下個 cycle 就切) |
| Context 儲存位置 | 記憶體 | 硬體 register file 常駐 |
| 排程者 | OS | 硬體 warp scheduler |
| 哲學 | 降低延遲 | 隱藏延遲 |
| 程式邏輯 | 各做各的 | SIMT,大量做同樣事 |

**核心洞察**:CPU 是「貴族」(少而精),GPU 是「人海戰術」(多而輕)。GPU 連 OS 都繞開,排程是硬體做的。

## 四、pthread 實作細節

### 基本流程
```c
pthread_t thread1;
pthread_create(&thread1, NULL, func, arg);  // 分裂執行流
pthread_join(thread1, NULL);                // 會合
```
- `pthread_create` 一呼叫就立刻分裂出新執行流
- `pthread_join` 強制 main 在此停下等待

### 為什麼 pthread 函式必須是 `void *func(void *)`
- C 沒有泛型,`void *` 是唯一能裝任意型別的固定大小型別
- 指標大小固定(ABI 友善)
- 避免回傳區域變數的生命週期問題
- `pthread_join` 的雙重指標 `void **` 設計是為了把回傳值寫回呼叫者

### Struct 打包慣用法
當需要傳多個參數時:
1. 定義 struct 包住所有參數
2. 呼叫端建立並填值
3. 傳 struct 位址給 `pthread_create`
4. Thread 函式內 cast 回 struct 指標
5. 用 `p->field` 存取

### 生命週期陷阱(重點)
**核心規則**:傳給 thread 的指標,所指向的資料必須在 thread 整個生命週期內保持有效。

- ✅ 安全:Heap(malloc)、global、static、有 join 的 stack
- ❌ 危險:函式 return 後仍被 thread 使用的 stack 變數

關鍵在於「**指標壽命不可超過資料壽命**」。`pthread_join` 之所以重要,就是它讓呼叫者的 stack frame 在 thread 用資料的期間保持存在。

## 五、⭐ 同步機制:Race Condition / Mutex / Atomic

### Race Condition 的本質
看似一行的 `total_count++`,在機器碼層級是 **3 個指令**:
```asm
load   $r0, [total_count]   ; 讀取
add    $r0, $r0, 1          ; 加一
store  $r0, [total_count]   ; 寫回
```

**任何一步都可能被打斷**。兩個 core 同時做 `++`,可能各自讀到舊值、各自加 1、各自寫回,結果**一次更新憑空消失**(lost update)。

### Race Condition 正式定義
> 「當一個 thread 正在寫某個記憶體位置,同時至少有另一個 thread 也在讀或寫同一個位置」就是 data race。

- ✅ 多人同時讀:OK
- ❌ 至少一方在寫:危險

### 為什麼這種 bug 特別陰險
1. **不一定每次都出錯**(timing 敏感,Heisenbug)
2. **規模放大才容易現形**(跑幾次看不出,跑大量才壞)
3. **環境換了就變**(換機器、加 log 就「修好」)
4. **程式碼看起來無辜**(`total_count++` 哪裡有問題?)

### Solution 1:Mutex(互斥鎖)
```c
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&mutex);    // 上鎖
total_count++;                  // 臨界區(Critical Region)
pthread_mutex_unlock(&mutex);   // 解鎖
```
- **同一時間只允許一個 thread 進入臨界區**
- 其他 thread `lock` 失敗時會 **block(進入 sleep)**
- OS 層級實作,通用但較慢
- 適合**複雜臨界區**、**多變數一起改**

### Solution 2:Atomic(硬體原子操作)
```c
__sync_fetch_and_add(&total_count, 1);
```
- CPU 用一條指令(如 x86 的 `lock addl`)在硬體層級保證**讀-改-寫不可分割**
- 比 mutex 快很多(沒有 syscall、沒有 context switch)
- 限於**單一變數**的簡單操作
- 重要家族:`fetch_and_add`、`fetch_and_or`、`compare_and_swap (CAS)`(lock-free 演算法基礎)

### 兩者比較
| 面向 | Mutex | Atomic |
|---|---|---|
| 實作層級 | OS / 函式庫 | CPU 硬體指令 |
| 效能 | 慢(可能 syscall) | 快(~10–30 ns) |
| 適用 | 任意大小臨界區 | 單變數簡單操作 |
| 等待時 | sleep,讓出 CPU | 通常 spin,不讓出 |

## 六、⭐ Monte Carlo 實戰:算 π

### 演算法原理
正方形 `[-1,1]²`(面積 4)內接圓 `x²+y²≤1`(面積 π)。隨機丟點:
$$\pi \approx 4 \times \frac{\text{圓內點數}}{\text{總點數}}$$

### 為什麼是平行化教學的黃金範例
1. **天然平行**:每次丟點獨立,沒有相依性(embarrassingly parallel)
2. **但需要同步**:`in_count` 是共享計數器,有 race condition
3. **結果好驗證**:答案接近 3.14159

### 四個版本的效能演進(以 1 億點 / 8 核心為例)

| 版本 | 同步機制 | 時間 | 加速比 | 主要瓶頸 |
|---|---|---|---|---|
| **1. 單執行緒** | 無 | 2.0 秒 | **1.0x** | 沒平行化 |
| **2. 多執行緒 + Mutex** | `pthread_mutex` | 30–60 秒 | **0.03–0.07x** ⚠️ | Lock contention、syscall |
| **3. 多執行緒 + Atomic** | `__sync_fetch_and_add` | 3–8 秒 | **0.25–0.7x** ⚠️ | Cache line ping-pong |
| **4. Local + Reduction** | 無(只在最後合併) | 0.27 秒 | **~7.4x** ✅ | 接近理論上限 |

### 三大教訓

**教訓 1:鎖的競爭可能毀掉所有平行性**
版本 2 比單執行緒慢 20 倍——「加 thread 應該變快」是錯覺。

**教訓 2:Atomic 不是萬靈丹**
雖然比 mutex 快,但**高頻寫入共享變數**會造成 cache line 在多核間瘋狂彈跳(ping-pong),L1/L2 cache 完全無用。

**教訓 3:減少共享才是王道**
最佳解法 = **每個 thread 用 private local 變數,結束時才合併**。這個 pattern 叫 **Reduction(歸約)**。

### Reduction Pattern(核心觀念)
```
壞模式:共享資源 ← 多人同時搶 → 變慢
好模式:資源切片 → 各自獨立工作 → 最後合併
```

業界應用:OpenMP 的 `reduction(+:x)`、CUDA reduction、MapReduce、Spark RDD、Rust Rayon——**都是同一個思想的變形**。

### 隱藏陷阱:`rand()` 不是 thread-safe
標準 `rand()` 內部有全域 seed 狀態,多 thread 呼叫會互相覆蓋,失去隨機性。要用 **`rand_r(&seed)`** 或 C++ `<random>`。

## 七、⭐ 平行化方法論(5 步驟通用心法)

```
1. Implement the serial version    → 先寫對,才能談快
2. Identify loops                  → 鎖定 for 迴圈(主要平行化機會)
3. Identify independent code       → 哪些 iteration 之間真正獨立?
4. Rewrite into threads            → 把獨立部分拆給多 thread
5. Aggregate the results           → Reduction!合併各 thread 結果
```

**判斷獨立性的方法**:問「如果打亂執行順序,結果會一樣嗎?」會 → 獨立;不會 → 有相依性。

### 補充第 6 條(投影片沒明說但極重要)
**6. Tune the thread count for your hardware** — 根據硬體調整 thread 數量

### 應用對照
- **OpenMP**:`#pragma omp parallel for reduction(+:x)` ← 自動做 step 4 + 5
- **CUDA**:thread 處理一個元素 + atomicAdd / shared memory reduction
- **MapReduce / Spark**:Map = step 4,Reduce = step 5

## 八、⭐ Thread 數量 vs 效能(U 型曲線)

### 兩張圖呈現的完整故事

**第一張(1–48 thread,16-core / 32-thread CPU)**

| 區域 | Thread 數 | 現象 | 原因 |
|---|---|---|---|
| Region 1 | 1–12 | **線性加速 (~11x)** | 每 thread 各自佔 physical core,理想 |
| Region 2 | 12–16 | 加速放緩 | 接近 core 上限,OS / cache / 記憶體頻寬競爭 |
| Region 3 | 16–32 | Hyperthreading 小幅加速 | 一 core 兩 thread 共用 ALU,只多 1.2~1.4x |
| Region 4 | 32+ | **效能反而變差** | 超過硬體能力,context switch 開銷反噬 |

**第二張(1–5000 thread,放大尺度)**

| 區域 | Thread 數 | 現象 |
|---|---|---|
| 1 | 1–50 | 急遽下降到甜蜜點(~0.07 秒) |
| 2 | 50–500 | 平台期,OS 還能應付 |
| 3 | 500–5000 | **線性上升到 0.18 秒,效能崩壞** |

### Thread 過多的隱形殺手

1. **Context switch 雪崩**:5000 個 thread 輪一圈要好幾秒,根本還沒輪到就結束
2. **Memory 爆炸**:每 thread 預設 8MB stack,5000 thread = 40GB 潛在佔用
3. **Cache 完全失效**:總 working set 遠超 cache 大小,hit rate 趨近 0
4. **Scheduler 自身開銷**:管理 5000 thread 的紅黑樹操作變慢
5. **OS 內部 lock contention**:scheduler lock、allocator lock 等
6. **NUMA 懲罰**:跨 socket 存取記憶體慢 2–3 倍

### 黃金法則

```
最佳 thread 數 ≈ 1 倍 ~ 1.5 倍的 hardware thread 數
```

- **CPU-bound**:`n_threads = n_physical_cores`
- **I/O-bound**:`n_threads = n_hardware_threads × 2~4`(大部分時間在等)
- **混合**:`n_threads = n_hardware_threads`

### 正確做法:Thread Pool

把「**任務數量**」和「**thread 數量**」**解耦**:
- 工作量 → 拆成「task」,放進 queue
- 硬體 → 開**固定數量**的 worker thread,從 queue 拿任務

各種框架的本質都是這個架構:Java `ExecutorService`、C++ TBB、Python `ThreadPoolExecutor`、Rust `rayon`、Go goroutine、OpenMP。

### 數學背景:Universal Scalability Law
$$\text{Speedup} = \frac{n}{1 + \alpha(n-1) + \beta n(n-1)}$$
- α:競爭(contention)
- β:一致性(coherence)成本

當 n 大到一定程度,$\beta n(n-1)$ 主導,加速比反而下降——這就是 Region 4 / 第二張圖的數學描述。

## 九、⭐ C++ Thread API(現代封裝)

### 核心訊息:`std::thread` 是 pthread 的 Syntax Sugar

底層在 Linux 上**就是呼叫 pthread**,但用 C++ template 包裝後**避免了所有 pthread 的醜陋之處**。

### 對照
| 痛點 | pthread | std::thread |
|---|---|---|
| 函式簽名 | 必須 `void *(*)(void *)` | **任何函式都可以** |
| 傳參數 | 只能 `void *` + struct + cast | **直接傳任意型別、任意數量** |
| 型別檢查 | 全靠手動 cast | **編譯時檢查** |
| 跨平台 | Windows 不原生支援 | **跨平台** |
| 忘了 join | UB | **`std::terminate()`,強制你正視** |

### Lock 的 RAII 包裝(解決「忘記 unlock」)

| 工具 | 特點 | 適用 |
|---|---|---|
| **`lock_guard`** | 最簡單,離開 scope 自動 unlock | 一般情況 |
| **`unique_lock`** | 可手動 lock/unlock、可轉移 | 配合 `condition_variable` |
| **`scoped_lock`** | 同時鎖多個 mutex(避免死鎖) | 多資源同步(C++17) |

範例:
```cpp
{
    std::lock_guard<std::mutex> lock(mutex);  // 建構時 lock
    // ... 做事 ... 即使中途 return / 拋例外也安全
}  // 離開 scope,自動 unlock
```

### 學習路徑階梯
```
1. pthread (理解底層) ← 你現在的位置 ✓
2. std::thread (寫實戰 C++)
3. std::async / std::future (任務思維)
4. OpenMP / TBB / Rayon (高階平行框架)
5. async/await / goroutine (語言原生並發)
```

每一層都建立在前一層之上。先學 pthread 不是浪費,是讓後面學什麼都通透的基礎。

## 思考脈絡(更新版)

整個學習過程的暗線是「**為什麼有這個抽象,它解決了什麼問題,代價是什麼**」:

- **ILP/TLP/DLP** → 不同層次的平行性如何取得
- **Process** → 解決共享記憶體的混亂,但帶來通訊成本
- **Thread** → 在隔離內找回共享,但帶來同步問題
- **pthread API 的怪異設計** → C 語言能力限制下的工程妥協
- **生命週期陷阱** → 共享帶來的責任(誰確保資料存在?)
- **Race condition** → 共享變數的根本困境
- **Mutex / Atomic** → 同步機制的兩種層次(OS / 硬體)
- **Monte Carlo 實戰** → 親眼見證「加 thread 反而變慢」的反直覺現象
- **Reduction pattern** → 「分散處理 + 集中合併」是平行運算的核心心法
- **方法論 5 步驟** → 把所有經驗濃縮成可套用的工程流程
- **Thread 數量曲線** → Thread 不是免費的,過多反而災難
- **C++ thread API** → 上層抽象解決下層醜陋,但理解下層才能用好上層

## 關鍵工程箴言

> 1. **「Premature optimization is the root of all evil」** — 先寫對,再寫快
> 2. **「加 thread 不一定變快,過頭反而變慢」** — Thread 數要匹配硬體,任務數要匹配工作量
> 3. **「指標壽命不可超過資料壽命」** — 共享記憶體的根本紀律
> 4. **「減少共享才是王道」** — Reduction 永遠勝過共享計數器
> 5. **「不要重複造輪子,但你應該知道輪子是怎麼做的」** — 用高階 API,但理解底層
